# 视频模块音频转码实现方案

## 一、需求总结

| 维度 | 决策 |
|------|------|
| 使用场景 | 浏览器兼容性转码 + 音质/格式统一 + 提取视频音轨 |
| 执行方式 | 按需转码（实时流） + 批量预转码 |
| 存储位置 | 视频同目录下的 `transcode/` 子目录 |
| 性能考虑 | 虚拟线程 + 可配置并发限制 + 跳过已转码文件 |

---

## 二、架构设计

### 2.1 新增文件

```
video/src/main/java/com/fryfrog/hub/video/
├── service/
│   └── AudioTranscodeService.java    # 核心转码服务
├── controller/
│   └── (VideoController 扩展端点)
└── dto/
    └── TranscodeRequest.java         # 请求DTO
```

### 2.2 核心服务 AudioTranscodeService

```java
@Service
public class AudioTranscodeService {
    private final String ffmpegPath;
    private final ExecutorService transcodeExecutor;
    private final int concurrencyLimit;
    private final Semaphore concurrencySemaphore;
}
```

**职责**：
1. 单文件音频转码
2. 批量转码（支持进度跟踪）
3. 获取转码后的音频流
4. 检查转码状态

### 2.3 转码流程

```
用户请求转码
  → 检查 transcode/ 目录是否已有文件 → 有则直接返回
  → ffprobe 分析源文件音频流信息
  → 构建 ffmpeg 命令
  → ProcessBuilder 执行转码
  → 保存到 {videoDir}/transcode/{baseName}-audio.{ext}
  → 返回结果
```



## 五、关键实现细节

### 5.1 FFmpeg 命令构建

```java
// 视频中的音频转码
ffmpeg -i input.mkv -vn -c:a aac -b:a 192k -ac 2 -ar 44100 output.m4a

// 参数说明：
// -vn           不处理视频流
// -c:a aac      音频编码器
// -b:a 192k     音频码率
// -ac 2         双声道
// -ar 44100     采样率
```

### 5.2 并发控制

```java
private final Semaphore semaphore;
private final ExecutorService executor;

public void transcodeAsync(Video video, TranscodeRequest request) {
    executor.submit(() -> {
        semaphore.acquire();
        try {
            doTranscode(video, request);
        } finally {
            semaphore.release();
        }
    });
}
```

### 5.3 跳过已转码

```java
Path getTranscodeOutputPath(Video video, String ext) {
    Path videoDir = Paths.get(video.getFilePath()).getParent();
    String baseName = getBaseName(video.getFileName());
    return videoDir.resolve("transcode").resolve(baseName + "-audio." + ext);
}
```

## 七、性能考量

| 机器配置 | 建议并发数 | 说明 |
|----------|-----------|------|
| 低配 (4核/8G) | 1 | 避免CPU/内存压力 |
| 中配 (8核/16G) | 2 | 平衡速度和资源 |
| 高配 (16核/32G) | 4 | 充分利用资源 |

通过 `hub.video.transcode.concurrent-limit` 配置，用户可根据机器性能调整。

### 虚拟线程优势
- 当前项目已启用 `spring.threads.virtual.enabled=true`
- ffmpeg 进程是 IO 密集型，虚拟线程非常适合
- 不占用操作系统线程资源

### 资源保护
- Semaphore 控制同时运行的 ffmpeg 进程数
- 每个 ffmpeg 进程有超时保护（默认30分钟）
- 转码失败不影响其他任务
