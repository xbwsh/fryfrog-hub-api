# Fryfrog Hub API — Performance Audit & Improvement Plan

## Executive Summary

Identified **16 performance bottlenecks** across all 4 media modules (video, music, comic, ebook). Grouped by severity:

- **Critical (4)**: N+1 queries, redundant API calls, `synchronized` blocking, unbounded `findAll()` in cleanup
- **High (5)**: Regex recompilation, in-memory grouping, ZIP re-scan per page, no batch lookups
- **Medium (5)**: No pagination, no connection pool tuning, missing DB indexes, format check overhead
- **Low (2)**: Unused code paths, watcher thread configuration

---

## Critical Issues

### C1. N+1 Query in `VideoController.toDTO()` — Per-Video Watch Progress

**File**: `video/src/main/java/com/fryfrog/hub/video/controller/VideoController.java:345-371`

```java
// Called per video in getAllVideos(), searchByTitle(), etc.
WatchProgress progress = watchProgressService.getProgress(video.getId());
```

**Impact**: 100 videos = 100 extra DB queries on every list endpoint.

**Fix**: Batch-fetch all `WatchProgress` records into a `Map<Long, WatchProgress>` before mapping.

---

### C2. Redundant TMDB API Calls in `saveActors()`

**File**: `video/src/main/java/com/fryfrog/hub/video/service/VideoService.java:920-952`

```java
// In scrapeAndBindTmdb(), detail is already fetched:
TmdbMovieDetail detail = tmdbService.getMovieDetail(tmdbId);  // 1st call
// ...
saveActors(saved, mediaType, tmdbId);
  // → saveOneActor → getMovieDetail(tmdbId) again  // 2nd call
```

**Impact**: 1 extra TMDB API call per video scrape. For batch scraping 100 videos = 100 wasted calls.

**Fix**: Pass the already-fetched detail object to `saveActors()`.

---

### C3. `synchronized` on `scrapeAndBindTmdb()` Blocks All Callers

**File**: `video/src/main/java/com/fryfrog/hub/video/service/VideoService.java:542`

```java
@Transactional
public synchronized Video scrapeAndBindTmdb(Long videoId, Long tmdbId, String mediaType) {
```

**Impact**: All scrape requests (manual + auto + sibling) are serialized. Under load, queue backs up.

**Fix**: Use per-video locking (e.g., `ConcurrentHashMap<Long, Lock>` or optimistic locking with `@Version`).

---

### C4. Unbounded `findAll()` in Cleanup Methods

**Files** (all 4 modules):
- `VideoService.cleanupInvalidRecords()` — line 182
- `MusicMetadataService.cleanupInvalidRecords()` — line 217
- `ComicMetadataService.cleanupInvalidRecords()` — line 200
- `EbookService.cleanupInvalidRecords()` — line 187

**Impact**: Loads entire table into memory. With 10k+ records, significant heap pressure and GC pauses.

**Fix**: Use `@Query` with `WHERE filePath IS NOT NULL` or stream with `ScrollableResults` / pagination.

## High Issues

### H1. Regex Patterns Recompiled Per Call

**Files**:
- `VideoService.parseSeasonEpisode()` — line 452-513: 7 regex patterns created per file
- `VideoService.cleanTitleForSearch()` — line 404-441: uses `DOTTED_QUALITY` + `QUALITY_PATTERN` (already static, good)
- `NfoService.cleanTitle()` — line 120-155: inline regex, not static
- `SeriesService.cleanTitle()` — line 146-181: inline regex, not static
- `ComicMetadataService.extractSeriesName()` — line 375-400: inline regex per call
- `ComicMetadataService.extractVolumeFromFileName()` — line 402-421: 5 regex patterns per call

**Impact**: Regex compilation is expensive. For batch operations (scanning 1000 files), this adds measurable overhead.

**Fix**: Extract all regex patterns to `private static final Pattern` fields.

---

### H2. `getComicsBySeries()` / `getEbooksBySeries()` — Full Table Scan + In-Memory Grouping

**Files**:
- `ComicMetadataService.getComicsBySeries()` — line 76-149
- `EbookService.getEbooksBySeries()` — line 77-116

```java
List<Comic> allComics = repository.findAll();
Map<String, List<Comic>> grouped = allComics.stream().collect(Collectors.groupingBy(...));
```

**Impact**: Loads all comics/ebooks, groups in Java. For 5000 comics, allocates many intermediate lists.

**Fix**: Add a `series` column index and use `GROUP BY` query, or at minimum use `findAll()` with `Specification` to filter.

---

```java

    try (ZipFile zipFile = new ZipFile(file)) {  // Opens, scans all entries, sorts, reads 1
```

**Impact**: Viewing page 50 of a comic = 50 ZIP opens + 50 full entry scans.

**Fix**: Cache the sorted entry list per comic file (e.g., `Map<String, List<ZipEntry>>` with size limit).

---

### H4. No Batch Lookups in `scanDirectory()` — N Queries for N Files

**Files**:
- `MusicMetadataService.scanDirectory()` — line 239-289: `findByFilePath()` per file
- `VideoService.extractAndSaveMetadata()` — line 205-267: `findByFilePath()` + `findByFileName()` per file

**Impact**: Scanning 500 music files = 500+ DB queries just for existence checks.

**Fix**: Pre-load all existing `filePath` values into a `Set<String>` at scan start. Use the set for O(1) lookups.

---

### H5. `isSupportedFormat()` Creates New Set Every Call

**Files**:
- `VideoWatcherService.isSupportedFormat()` — line 192-196
- `MediaWatcherService.isSupportedFormat()` — line 180-184

```java
Set<String> formats = Set.of(supportedFormats.split(","));  // new Set per event
```

**Impact**: Minor but wasteful — called on every file system event.

**Fix**: Cache the parsed set in a field, initialized in `@PostConstruct`.

## Medium Issues

### M1. No Pagination on List Endpoints

**Files**: `VideoController.getAllVideos()`, `MusicController`, `ComicController`, `EbookController`

All use `findAll()` without pagination. With 10k+ media files, response is huge.

**Fix**: Add `Pageable` support via Spring Data `PagingRepository` or `@Query` with LIMIT.

---

### M2. HTTP Connection Pool Not Tuned

**File**: `common/src/main/java/com/fryfrog/hub/common/config/RestTemplateConfig.java`

Uses `SimpleClientHttpRequestFactory` — creates a new connection per request. No connection pooling.

**Fix**: Use `HttpComponentsClientHttpRequestFactory` with Apache HttpClient connection pool for TMDB/scraper calls.

---

### M3. Missing Database Indexes

No custom indexes beyond JPA auto-generated ones. Frequently queried fields:
- `Video.filePath` (unique — good)
- `Video.title` (LIKE queries — needs index)
- `Video.tmdbId` (frequently filtered)
- `MusicTrack.filePath` (unique — good)
- `MusicTrack.artist` (LIKE queries)

**Fix**: Add `@Index` annotations or Liquibase/Flyway migration for critical query paths.

---

### M4. `autoGroupSeries()` Loads All Videos

**File**: `video/src/main/java/com/fryfrog/hub/video/service/VideoService.java:335-376`

```java
List<Video> allVideos = repository.findAll();
```

**Impact**: Groups all videos in memory to find ungrouped ones.

**Fix**: Add `findBySeriesIsNull()` query to only load ungrouped videos.

---

### M5. `cleanupDuplicateSeries()` N+1 on Series Lookup

**File**: `video/src/main/java/com/fryfrog/hub/video/service/SeriesService.java:234-266`

```java
for (Long tmdbId : duplicateTmdbIds) {
    List<VideoSeries> duplicates = seriesRepository.findByTmdbId(tmdbId)...
```

**Fix**: Batch-load all duplicate series in one query.

---

## Low Issues

### L1. `MusicScrapeService.scrapeAll()` Filters in Java

**File**: `music/src/main/java/com/fryfrog/hub/music/service/MusicScrapeService.java:83-110`

```java
List<MusicTrack> tracks = repository.findAll();
List<MusicTrack> needScrape = tracks.stream().filter(this::needsScraping).toList();
```

**Fix**: Add `findByScrapeStatusNot("scraped")` or use `@Query` with conditions.

---

### L2. `VideoController.streamVideo()` Doesn't Implement Range Requests

**File**: `video/src/main/java/com/fryfrog/hub/video/controller/VideoController.java:175-191`

Returns full file despite setting `Accept-Ranges: bytes`. No actual range handling.

**Fix**: Implement `ResourceRegion` or use `Range` header parsing for proper streaming.

---

## Recommended Implementation Order

| Phase | Items | Effort | Impact |
|-------|-------|--------|--------|
| 1 | C1 (N+1 in toDTO), C2 (redundant API), C4 (findAll cleanup) | Low | High |
| 2 | H1 (regex), H4 (batch scan), H5 (format cache) | Low | Medium |
| 3 | H3 (ZIP cache), M4 (autoGroupSeries query) | Medium | Medium |
| 4 | M1 (pagination), M2 (connection pool), M3 (indexes) | Medium | Medium |
| 5 | C3 (synchronized), M5 (series N+1), L1 (scrape filter), L2 (range) | High | Medium |
